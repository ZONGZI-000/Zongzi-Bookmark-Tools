# 粽子书签工具需求书

## 1. 产品定位

粽子书签工具是一款围绕浏览器书签管理的长期维护型 Chrome/Chromium 扩展。
核心目标不是单一的 WebDAV 同步，而是提供一套完整的书签工具能力，覆盖：
- 书签检索
- 书签摘要
- 书签同步
- 书签清理
- 对应设置管理

产品默认以中文界面为主，强调安全、保守、可回退，避免误删和误覆盖。

## 2. 模块划分

### 2.1 书签检索
目标：让用户打开 popup 后就能快速检索本地书签。

当前要求：
- popup 初次打开时，默认先获取本地全部收藏夹中的书签基础信息。
- 预取字段至少包含：
  - 名称
  - 网址
  - 文件夹路径
- popup 默认进入“书签检索”tab。
- 检索支持按标题、URL、地址、文件夹路径，以及后续摘要相关内容进行匹配。
- 若预热失败，仍允许回退到现有后台检索链路。

### 2.2 书签摘要
目标：通过离线引擎或 AI 引擎，为书签生成结构化摘要信息，提升检索能力。

当前要求：
- popup 中只保留：
  - 自动索引开关
  - 摘要引擎设置入口
  - 打开摘要列表入口
  - 摘要状态
- 自动索引开启时，默认同时启用自动离线索引与自动 AI 索引的后台策略。
- 摘要引擎支持：
  - TextRank 离线引擎
  - 本地小模型（Transformer.js）
  - Chrome Gemini Nano
  - 自定义 OpenAI
  - 自定义 Anthropic
- 本地小模型支持：
  - 多模型独立存储
  - 纯本地文件夹一键导入（不含远程下载）
  - 提供 HuggingFace / hf-mirror / ModelScope 镜像站链接，用户手动下载模型文件夹后导入
  - 导入结果展示（已导入文件列表及大小、缺失文件、错误详情）
  - 删除模型缓存
  - 切换模型时自动更新导入指引和下载地址
  - 导入完成状态展示：模型已就绪（xx MB）/ 已导入 x/5 个文件 / 未导入
  - IndexedDB 存储结构：`zongzi_model_store` / `modelFiles` + `modelFileMeta`，按 `模型ID::文件路径` 分键存储
  - 支持 webkitdirectory 文件夹选择器，自动匹配文件路径
- 摘要列表支持：
  - 查看名称 / 地址 / 文件夹 / 分类 / 关键词 / AI 摘要等
  - 多选
  - 批量离线摘要
  - 批量 AI 摘要
  - 编辑单条记录
  - 手工刷新状态

### 2.3 书签同步
目标：通过多种后端进行书签备份、恢复、智能同步，并优先保证无网络备份能力。

当前要求：
- 默认同步方式为本地文件导入/导出，便于用户开箱即用地做本地备份与恢复。
- 支持上传备份、下载恢复、智能同步。
- 同步后端采用 provider 抽象，WebDAV、本地文件、GitHub、Gitee 统一接入同一套快照读写流程。
- 当前可用后端：
  - 本地文件导入/导出：支持手动导出 JSON 备份和从 JSON 文件导入恢复，不依赖网络。
  - WebDAV：支持自动同步、上传备份、下载恢复、智能同步。
  - GitHub：通过 Contents API 读写一个书签 JSON 文件，提交记录作为版本历史。
  - Gitee：通过 Contents API 读写一个书签 JSON 文件，提交记录作为版本历史。
- GitHub / Gitee 设置项包含 token、用户名/组织、仓库名、分支、文件路径；token 需本地加密存储。
- 未配置当前同步后端时，对应上传/下载/智能同步和自动同步能力必须受限，不能误操作；本地文件导入/导出不受 WebDAV 配置限制。
- popup 同步按钮必须按当前同步方式展示：WebDAV / GitHub / Gitee 展示智能同步、上传、下载；本地文件只展示导出文件、导入文件，避免出现不适用操作。
- 默认文案和设置提示应以“多同步方式”作为主视角，不再默认展示为 WebDAV；仅在 WebDAV 设置和测试链路中保留 WebDAV 专属提示。
- 同步方式展示顺序统一为：本地文件导入/导出 → WebDAV → GitHub → Gitee。
- 下载模式支持：
  - 安全模式
  - 完全镜像模式
- 安全模式默认更保守，不能破坏原有收藏夹内容。
- WebDAV 配置、Git/Gitee 令牌、自定义 AI API Key 需统一进入本地加密存储，不在普通 settings 明文字段中持久化。
- 设置页可临时查看/隐藏 WebDAV 密码、Git/Gitee token 和自定义 AI API Key，但默认均以密码框隐藏。
- 报错、连接测试、同步失败、AI 验证失败等场景必须对密码、token、API Key、URL query 中的敏感参数做脱敏。
- 同步设置页只保留后端配置、连接测试和本地文件导入导出；自动同步、同步间隔、下载模式等运行时设置统一只在 popup 外部面板修改，不在 options 页重复提供。

### 2.4 书签清理
目标：帮助用户清理失效书签及本地冗余结构。

当前要求：
- popup 内分为：
  - 清理失效书签入口
  - 本地书签整理入口
- 失效书签清理独立页面支持：
  - 扫描
  - 暂停 / 继续
  - 进度展示
  - 结果分类
  - 多选删除
  - 删除确认
  - 清理记录
- 本地整理能力支持：
  - 清理重复书签
  - 清理空文件夹
  - 一键清理

## 3. Popup 页面结构

popup 当前目标顺序为：
1. 书签检索
2. 书签摘要
3. 同步书签
4. 清理书签

设计原则：
- 检索优先
- 摘要其次
- 同步与清理保留明确入口
- popup 不承载复杂设置表单，只承载高频操作和设置入口

## 4. 设置页结构

插件选项页统一为两个 tab：

### 4.1 同步设置
包含：
- 同步方式选择：本地文件导入/导出、WebDAV、GitHub、Gitee
- WebDAV 文件夹地址
- 用户名
- 密码 / 应用令牌
- 远端文件名
- 请求超时
- WebDAV 测试
- GitHub / Gitee 连接测试
- 本地文件导出 / 导入

说明：
- 自动同步、同步间隔、下载模式属于 popup 同步面板中的运行时设置，不在设置页重复提供修改入口。

### 4.2 摘要引擎设置
包含：
- 引擎选择
- 本地小模型下载与管理
- Gemini Nano 设置引导与验证
- 自定义 OpenAI 配置与验证
- 自定义 Anthropic 配置与验证
- 摘要相关状态

补充规则：
- popup 中所有设置入口统一进入 `options.html`
- `summary-settings.html` 不再作为主设置入口，可仅做兼容跳转

## 5. 数据流与后台任务

### 5.1 检索预热
- popup 打开时，后台读取本地全部书签基础信息。
- 复用现有 `chrome.bookmarks.getTree()` 采集链路。
- 前台缓存用于首次检索命中。

### 5.2 自动索引
- 监听新增书签事件。
- 自动将书签加入摘要处理队列。
- 根据配置决定使用离线引擎、AI 引擎或回退策略。

### 5.3 模型管理
- 本地小模型文件保存在 IndexedDB（`zongzi_model_store`，stores: `modelFiles` + `modelFileMeta`）。
- 不同模型按 `模型ID::文件路径` 分开存储。
- 模型导入方式为纯本地文件夹导入：用户先从镜像站下载模型文件夹到本机，然后在设置页点击"选择模型文件夹导入"一键导入全部文件。
- 支持 3 个内置模型：LaMini-Flan-T5 783M、mT5-small、DistilBART CNN 6-6。
- 导入时自动根据文件名和相对路径匹配合并模型文件到 IndexedDB。
- 状态包括：未导入、部分导入、已完成、删除完成等。
- 后台与前台都需要按逐文件真实缓存状态判断 ready，不能只按文件条数粗略判断。

## 6. 默认行为与安全策略

- 默认语言：中文
- 默认同步策略：安全、保守
- 自动索引开启后，应按统一后台契约运行，避免 UI 与真实行为不一致
- 未配置当前同步后端时，不允许执行对应网络同步操作；本地文件导入/导出作为离线手动备份能力保持可用
- 删除、镜像覆盖、自动清理等高风险动作要谨慎提示
- 敏感配置统一加密存储：WebDAV URL / 用户名 / 密码、GitHub/Gitee token、自定义 AI API Key。
- 兼容历史版本：若发现旧版本明文字段，应自动迁移进加密配置，并从普通 settings 中移除。
- UI 默认隐藏敏感字段；用户主动点击“查看”时仅临时明文展示，切换语言或重开页面不应默认展开。

## 8. 待持续优化问题

- 检索预热后的体验与性能仍需回归验证
- 设置页合并后要继续检查入口是否统一、是否仍有旧路径残留
- 摘要引擎状态、模型状态、AI 验证状态要持续校验中英文展示
- 自动索引与摘要列表的状态同步要避免串台或展示滞后
- 敏感配置加密使用浏览器本地扩展 ID 派生密钥，主要防止普通存储明文暴露；不等同于操作系统级密钥保险箱，后续如发布商店版可继续评估更强的凭据隔离策略。
- 本地模型导入为纯离线操作，不涉及 HTTP Range 断点续传。

## 9. 本次变更记录（2026-05-17）

本次需求重点（模型缓存 IndexedDB store 缺失修复）:
- 用户最新报错为 `Failed to execute 'transaction' on 'IDBDatabase': One of the specified object stores was not found`，触发点发生在本地模型下载第一条文件的“校验文件/写入元信息”阶段。
- 根因是部分旧安装环境中的 `zongzi_model_store` 只创建了 `modelFiles`，缺少 `modelFileMeta`；但设置页下载器、旧独立下载页、后台状态读取、offscreen 推理对数据库版本和升级路径不一致，导致某些入口能读取、某些入口一 transaction 就报错。
- 本轮统一把模型缓存库 schema 升到 v3，并在 `options.js`、`model-download.js`、`background.js`、`offscreen-inference.js` 中对齐升级逻辑，确保旧库会自动补建 `modelFileMeta`。
- 设置页下载流程中的 `readModelFileBuffer()` 也改为复用统一 `openDB()`，避免继续用旧版本号直接打开旧库，导致下载阶段落入缺 store 的旧 schema。

## 12. 上次变更记录（2026-05-17）

本次需求重点（模型下载状态收敛与续传安全修复）:
- 设置页内联下载器、旧 `model-download.html/js` 独立下载页、后台 `computeLocalModelCacheState()` 的完成态口径继续收敛：显式 `status = complete` 且有真实缓存字节时，视为权威完成态；不再只依赖 `expectedSize > 0`。
- 镜像 `etag` / `last-modified` / `Range` 不满足安全续传条件时，不再在预检查阶段把已有缓存直接清空，而是标记为 `stale` 并保留旧缓存，等确认新响应可用后再决定是否整文件重下。
- 当 resume 请求发出后服务端返回 `200` 而不是 `206` 时，本次请求按“从头下载”处理，但不会在请求前擦掉旧缓存，降低第一条文件反复回到重下的概率。
- 下载成功后的最终 `expectedSize` 允许使用真实 `buffer.byteLength` 回写，避免某些镜像不给稳定 HEAD/Range 信息时，文件虽然下完了却长期无法被识别为完成。
- 旧独立下载页与设置页下载器继续保持同一套镜像 fallback、缓存恢复和完成判定口径，避免两个入口行为再次分叉。

## 14. 上次变更记录（2026-05-17）

本次需求重点（本地模型下载完成判定修复）:
- 设置页内联模型下载器和旧 `model-download.html/js` 独立下载页，文件完成判定统一改为“IndexedDB 中真实 `byteLength >= 预期文件大小` 才算已完成”，不再只按文件存在性跳过。

- 内联下载器删除模型后会同步清空 `fileProgressMap` 和 `fileLoadedByteMap`，避免删除后仍残留旧进度 UI。
